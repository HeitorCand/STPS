The Indexer is the service that receives webhooks, parses transactions, and
emits normalized events to the Scoring Engine.

It exists to keep the rest of the system from being coupled to raw chain or
Helius payload formats.

## Parsers

The Indexer includes parsers for:

- Squads;
- SPL Governance;
- System nonce.

Each parser tries to recognize a risk event and return a `GovernanceEvent`.

This cleanly separates two responsibilities:

- the parser understands the shape of the transaction;
- the Scoring Engine understands the impact of the event.

## Normalized event

Logical shape:

```ts
type GovernanceEvent = {
  type:
    | "MULTISIG_THRESHOLD_CHANGED"
    | "TIMELOCK_CHANGED"
    | "SIGNER_ADDED"
    | "SIGNER_REMOVED"
    | "EMERGENCY_KEY_USED"
    | "NONCE_ACCOUNT_CREATED"
    | "NONCE_ADVANCED";
  protocolAddress: string;
  sourceProgram: "squads" | "spl-governance" | "system-nonce";
  rawSignature: string;
  timestamp: number;
  metadata: Record<string, unknown>;
};
```

This intermediate contract is a central STPS building block. It is what allows
you to swap parsers, add new sources, or change ingestion logic without
rewriting the entire scoring engine.

## Webhook security

When a webhook secret is configured, the Indexer requires that secret in the
request path used by the upstream sender.

## Why this matters to the product

Without an indexing layer, every integration would have to understand raw
payloads, program IDs, and different governance formats. The Indexer removes
that complexity from the public product surface and makes the system more
predictable for both the dashboard and the SDK.
