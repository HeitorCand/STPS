The Scoring Engine is the main STPS backend. It calculates protocol score and
also serves the authenticated operator workspace.

## Responsibilities

- receive normalized events;
- recalculate score, risk level, flags, and history;
- register protocols when needed;
- authenticate wallet sessions;
- store claims and verification state;
- issue persistent SDK tokens;
- write certificate updates to the Anchor program.

## What it coordinates

The Scoring Engine plays four roles at once:

- it receives already-normalized events;
- it calculates score, flags, and history;
- it maintains the private per-account workspace;
- it issues persistent tokens for SDK consumption.

In practice, this is the point where STPS technical logic becomes product.
The Indexer delivers signals. Anchor persists state. But the Scoring Engine is
the layer that decides what a protocol currently is in operational trust terms.

## How it thinks about risk

The engine does not treat score as an isolated number. It treats score as a sum
of causes:

- an event arrives;
- the engine identifies the risk category;
- flags are activated;
- history receives a readable reason;
- the final certificate is updated.

This matters because the value of STPS is not just saying a protocol is at `42`.
The value is being able to explain why that `42` happened.

## Governance events

The Scoring Engine understands events such as:

- `MULTISIG_THRESHOLD_CHANGED`
- `TIMELOCK_CHANGED`
- `SIGNER_ADDED`
- `SIGNER_REMOVED`
- `EMERGENCY_KEY_USED`
- `NONCE_ACCOUNT_CREATED`
- `NONCE_ADVANCED`

These events are not useful in isolation. The job of the engine is to interpret
their operational impact in protocol context and turn them into readable flags
and score changes.

## Operator workspace

In addition to score, the Scoring Engine also supports the account identity
layer of the product. That includes:

- operator sessions;
- protocol claims;
- verification status;
- issuing and revoking persistent tokens.

This reduces fragmentation. Instead of separate “risk backend” and “account
backend” systems, STPS keeps account state close to certificate state because
they belong to the same product flow.

## Supabase

The schema defines:

- users;
- wallet identities;
- auth challenges;
- sessions;
- protocol claims;
- api tokens.

The database exists to support relationships and access, not to replace the
chain. Claims, sessions, and tokens live in the workspace. The certificate
remains the central trust object of the product.
