## Surfaces

| Component | Folder | Responsibility |
| --- | --- | --- |
| Indexer | `packages/indexer` | Receive webhooks and normalize events. |
| Scoring Engine | `packages/scoring` | Compute score, flags, history, and private workspace state. |
| Anchor Program | `programs/stps` | Persist the certificate on-chain. |
| Dashboard | `frontEnd/stps-client` | Login, claim, verification, tokens, and operational reading. |
| SDK | `packages/sdk` | Authenticated access to account-linked protocols. |

## What each block solves

### Indexer

The Indexer exists to isolate chain noise from the rest of the product. Real
transactions are heterogeneous, full of program-specific formats and
instructions. The Indexer turns that into a small, predictable, stable internal
contract. That keeps the Scoring Engine from having to understand every raw
Helius payload or every parsing edge case.

### Scoring Engine

The Scoring Engine is the decision layer. It takes a normalized event and
answers three questions:

- does this change the trust state of the protocol?
- which flags explain the change?
- how should that new state be exposed in the workspace and SDK?

This is the layer that gives the product coherence because it centralizes
score, explainability, history, and operator account identity.

### Anchor Program

The Anchor program gives the certificate persistence and verifiability. Instead
of existing only as an API response, the score also becomes protocol state with
on-chain semantics. That matters to the STPS thesis because it moves the
product closer to a certificate authority for Solana operations.

### Dashboard

The dashboard is the human operational surface. It does not exist to show a
generic public catalog. It exists to serve the team operating a protocol. That
is where trust logic becomes workflow: claim, verification, score reading,
certificate inspection, and token issuance for integrations.

### SDK

The SDK is the automation surface. It takes the same state the team sees in the
dashboard and carries it into bots, internal services, CI pipelines, and
monitors. Its role is not to recalculate risk. Its role is to propagate the
trust state STPS has already decided to other systems.

## Core data model

The central object is the protocol certificate:

- `protocolAddress`
- `currentScore`
- `riskLevel`
- `activeFlags`
- `riskFlagsBitmask`
- `lastUpdate`
- `history`

That state can be shown in the dashboard, read by the SDK, and persisted by the
Anchor program.

## Score flow

1. A governance event happens on Solana.
2. Helius sends the payload to the Indexer.
3. The Indexer converts the payload into a `GovernanceEvent`.
4. The Scoring Engine recalculates score and flags.
5. The dashboard and the SDK read the updated state.

The important point is that the score does not originate from a manual user
call. It originates from operational protocol events. The dashboard and the SDK
consume an already consolidated state; they are not the origin of that state.

## Operator flow

1. A wallet signs the challenge.
2. The backend creates a session.
3. The operator claims the protocol.
4. The backend verifies control.
5. The operator creates persistent tokens for integrations.

This flow separates two ideas that are often mixed together:

- who controls the protocol as the operator;
- who consumes the trust state as a system.

The wallet solves the first. The persistent token solves the second.

## On-chain writing

The Scoring Engine registers and updates certificates through Anchor as a normal
part of the product flow.
