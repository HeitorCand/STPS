The current dashboard is not a global public catalog. It is a private
account-based workspace accessed through wallet signatures.

## Login

Login uses a wallet-signed challenge:

1. The client requests a challenge from the backend.
2. The wallet signs the message.
3. The client sends the signature back.
4. The backend creates a session.

## Protocol claim

A claim links a `programId` to the operator account.

Core fields:

- `protocolAddress`
- `label`
- `claimedByWallet`
- `status`
- `verificationMethod`
- `verificationTarget`

## Control verification

The current strong path accepts:

- `upgrade_authority`: the signing wallet is the program upgrade authority;
- `known_admin_signer`: the wallet belongs to the admin/signer set known by STPS.

When there is not enough automatic proof, the claim moves to `manual_review`.

## SDK tokens

Tokens are persistent and tied to the account. They are not session tokens.

Properties:

- they do not expire automatically;
- they can be revoked;
- they are stored in the database only as hashes;
- the raw value is shown only at creation time.
